01 · Management overview
Co PrintGuard řeší
01

PrintGuard propojuje provozní data z tisku, skladu, checklistů a externích systémů do jednoho interního nástroje pro rychlé rozhodování a kontrolu výroby.

Stock & inventory

Pohyby materiálu, stavy skladu, alerty na kritické položky a audit změn.

Print log

Colorado CSV/ACL accounting, délky tisku, job statusy a historický reporting.

Submit / processed orders

Processed XML z NAS, pipeline orderů, reprinty a vazba na původní zakázku.

Checklist & push

Provozní checklisty, splnění úkolů, připomínky a push notifikace.

Praktický efekt
  • méně ručního dohledávání v NAS / Submit Tool / Onyx / Colorado logách
  • lepší kontrola, jestli se order poslal, zpracoval a vytiskl
  • auditní stopa pro checklisty a reprinty
  • nižší riziko provozní slepoty při chybách v print procesu

Současný charakter systému

PrintGuard už není jednoduchý prototyp. Je to interní provozní systém napojený na databázi, NAS, Netlify Functions, Windows Task Scheduler, Post Purchase API, Submit Tool logy a Colorado accounting.

Doporučený režim vývoje: feature freeze pro nové nápady, prioritně stabilizace, dokumentace, env cleanup, DB governance a release workflow.

02 · Technická architektura
Runtime shape
03

Frontend

Classic-script vanilla JS PWA. Shell je v index.html, orchestraci stále částečně drží app.js, nové části jsou v app/, scripts/ a reports/.

Backend

Netlify Functions pro HTTP API. Sdílené helpery jsou v netlify/functions/_lib/.

Database

Neon PostgreSQL. Persistují se sklad, Colorado záznamy, print accounting, processed ordery, push subscriptions a checklist audit.

Server pipelines

Windows/Node/PowerShell skripty pro NAS, Submit Tool, Post Purchase a Colorado accounting.

03 · Konfigurace a zabezpečení
Security posture
07

Dobře

  • DB connection není ve frontendu
  • backend operace jdou přes Netlify Functions
  • VAPID private key zůstává server-side
  • admin endpointy mají server-side guard

Slabší místa

  • není OAuth / Entra ID
  • admin PIN není enterprise identity
  • legacy DB aliasy můžou ukazovat na starou DB
  • service worker cache může mást při deployi
Minimální provozní pravidla
  • Netlify ENV držet ve firemním účtu
  • Frankfurt Neon DB je canonical production DB
  • všechny DB aliasy musí ukazovat na stejnou DB, dokud nejsou odstraněné
  • po migraci sledovat starou DB, jestli se do ní stále nezapisuje
  • nepouštět scheduled tasky pod účtem bez NAS přístupu

04 · Provozní runbook
Cutover / migration checklist
10
Model: jedna production DB. Po přepnutí na Frankfurt DB nesmí aktivní klienti a tasky zapisovat do staré Ohio DB.
  1. ověřit restore dat do nové Frankfurt DB
  2. přepsat Netlify ENV: NEON_DATABASE_URL + legacy aliasy
  3. redeploy Netlify s clear cache
  4. přepsat Windows Machine ENV na print serveru
  5. zkontrolovat .env soubory mimo Git
  6. restartovat scheduled tasky / terminály
  7. spustit smoke test v appce
  8. sledovat starou Ohio DB, jestli se už neaktualizuje
PowerShell kontroly
